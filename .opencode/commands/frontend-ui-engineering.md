# Frontend UI Engineering

Build production-quality user interfaces that are accessible, performant, and visually polished. The goal is UI that looks like it was built by a design-aware engineer — not generated by an AI.

## When to Use

- Building new UI components or pages
- Modifying existing user-facing interfaces
- Implementing responsive layouts
- Adding interactivity or state management
- Fixing visual or UX issues

## Key Standards

### Avoid the AI Aesthetic
- No purple/indigo everything
- No excessive gradients
- No rounded-2xl everywhere
- No generic hero sections
- No lorem ipsum-style copy
- No shadow-heavy design

### Accessibility (WCAG 2.1 AA)
- Every interactive element is keyboard accessible
- ARIA labels on elements without visible text
- Focus management for dynamic content
- Meaningful empty and error states
- Sufficient color contrast (4.5:1 for text)

### Component Architecture
- Colocate everything related to a component (tsx, test, stories, hooks, types)
- Prefer composition over configuration
- Keep components focused (split if >200 lines)
- Separate data fetching from presentation

### State Management Hierarchy
- Local state (useState) → Component-specific UI state
- Lifted state → Shared between 2-3 sibling components
- Context → Theme, auth, locale (read-heavy, write-rare)
- URL state → Filters, pagination, shareable UI state
- Server state (React Query, SWR) → Remote data with caching
- Global store (Zustand, Redux) → Complex client state shared app-wide

### Responsive Design
- Mobile first, then expand
- Test at: 320px, 768px, 1024px, 1440px

## Trigger

"Build this UI: [description]"
"Create a component: [spec]"
"Fix the UI for: [issue]"

## Output

Production-quality UI with accessibility, responsive design, proper state management, and no generic "AI aesthetic."
